## 1. Problem

Technical interview preparation is hard not because candidates can't solve problems, but because they can't practice the full experience on demand. Practising with another person is logistically difficult. Practising alone leaves the most critical skill untrained: explaining your reasoning out loud under pressure.

Research by Behroozi and Barik found that candidate performance in technical interviews drops by more than 50% when solving problems under direct observation. The anxiety is real, and the only way to reduce it is repetition.

This platform exists to make that repetition accessible — at any time, for any level, without needing another person in the room.

---

## 2. Product Vision

A web-based mock interview platform that simulates the structure of a real technical interview in two phases: behavioral questions answered by audio, followed by an algorithm problem explained in text. An AI evaluates both phases and produces a structured feedback report.

The core design principle: **evaluate the explanation and reasoning process, not code execution.** This is what real interviewers care about.

---

## 3. Users

Primary user: a software engineering candidate (student or professional) preparing for technical interviews.

They want to:
- Practice in a low-stakes environment before real interviews
- Hear honest, structured feedback on how they communicate
- Identify specific areas to improve, not just "you got it wrong"
- Do this alone, on their own schedule

---

## 4. Scope

### In scope (MVP)
- 3 behavioral questions answered via audio recording
- 1 LeetCode-style algorithm question answered via text
- AI transcription and evaluation of all responses
- Final structured feedback report
- Bilingual interface: Portuguese and English
- Three difficulty levels: Junior, Mid, Senior
- Fully serverless — no account creation, no database

### Out of scope
- Code execution or automated test running
- Facial expression or tone-of-voice analysis
- Persistent user accounts or history across sessions
- Multiple simultaneous coding questions (single question per session)
- Mobile-native app

---

## 5. Interview Flow

```
Landing
  → user enters name, selects level and language

Behavioral Phase (3 questions)
  → AI presents question
  → candidate records audio response
  → Whisper transcribes audio server-side
  → GPT-4o evaluates transcript
  → scores saved to session; next question loads

Transition
  → brief summary of behavioral performance
  → candidate proceeds to coding phase

Coding Phase (1 question)
  → GPT-4o generates a LeetCode-style question at the selected difficulty
  → candidate types their explanation and approach
  → GPT-4o evaluates the written response

Feedback
  → comprehensive report rendered from all session data
```

## 6. Functional Requirements

### 6.1 Landing Page

- Candidate enters their name (text input)
- Candidate selects difficulty level: Junior, Mid, or Senior
- Candidate selects language: PT or EN
- Upon submission, 3 behavioral questions are randomly selected from a bank of 8 and stored in session
- Session is initialised in `sessionStorage`

### 6.2 Behavioral Phase

- Each question is displayed one at a time
- Candidate records audio using the browser's `MediaRecorder` API
- Recording states: idle → recording → recorded
- Candidate may re-record before submitting
- On submit: audio sent to `/api/behavioral` as `multipart/form-data`
- Server transcribes with `whisper-1`, then evaluates with `gpt-4o`
- Scores (1–5) returned for: STAR structure, clarity, relevance
- Coaching feedback string also returned
- Result saved to session; next question loaded

**Behavioral question bank (8 questions, 3 selected randomly per session):**

1. Tell me about a recent technical challenge you faced and how you overcame it.
2. Describe a time you had to work under pressure to deliver on a deadline.
3. Tell me about a project you're proud of and your specific contribution.
4. How do you handle disagreements with a teammate about a technical approach?
5. Describe your debugging process when you can't immediately find the cause of a bug.
6. Tell me about a time you had to explain something technical to a non-technical person.
7. What's the most valuable lesson you've learned from a mistake at work or in a project?
8. How do you prioritize when multiple tasks are competing for your attention?

### 6.3 Coding Phase

- A LeetCode-style algorithm question is generated by `gpt-4o-mini` based on level and language
- Difficulty mapping: Junior → easy, Mid → medium, Senior → hard
- Question format includes: title, description, 2–3 examples, constraints, expected time/space complexity
- Candidate types their explanation and approach in a textarea (no code execution)
- On submit: sent to `/api/behavioral` for evaluation by `gpt-4o`
- Scores (1–5) returned for: problem understanding, approach quality, technical accuracy, edge cases, communication

### 6.4 Feedback Report

- Rendered from full session data via `/api/feedback`
- Contains:
  - Overall score (1–5), weighted 40% behavioral / 60% coding
  - Verdict: "Strong hire", "Hire with reservations", or "Needs more preparation"
  - Behavioral summary (per-question scores + coaching note)
  - Coding summary (scores + feedback)
  - Top 3 strengths
  - Top 3 areas to improve
  - Study recommendations (3 topics)
  - Final note — personal, encouraging closing

## 7. Technical Architecture

### Stack

| Layer | Technology | Reason |
|---|---|---|
| Framework | Next.js (App Router) + TypeScript | Serverless API routes, SSR, Vercel-native |
| Styling | Tailwind CSS | Utility-first, no runtime overhead |
| AI — transcription | OpenAI `whisper-1` | Only production-grade browser audio transcription |
| AI — evaluation | OpenAI `gpt-4o` | Required reasoning quality for nuanced feedback |
| AI — question gen | OpenAI `gpt-4o-mini` | Cheaper, sufficient for structured generation |
| Session state | Browser `sessionStorage` | True session memory — cleared on tab close, no server |
| Hosting | Vercel | Serverless, zero-config Next.js deployment |

### Pages

| Route | Purpose |
|---|---|
| `/` | Landing — name, level, language |
| `/interview` | Interview state machine (behavioral + coding) |
| `/feedback` | Final report |

### API Routes (serverless)

| Endpoint | Input | Output |
|---|---|---|
| `POST /api/behavioral` | `audio: File`, `question: string`, `level`, `language` | `{ transcript, scores, strengths, improvements, feedback }` |
| `POST /api/question` | `level`, `language` | `{ title, description, examples, constraints, topic, expectedComplexity }` |
| `POST /api/feedback` | Full session object | `{ overallScore, verdict, behavioralSummary, codingSummary, topStrengths, topImprovements, studyRecommendations, finalNote }` |

### Session State Schema

```typescript
interface InterviewSession {
  candidateName: string
  level: 'junior' | 'mid' | 'senior'
  language: 'pt' | 'en'
  behavioralQuestions: string[]
  behavioralResults: Array<{
    question: string
    transcript: string
    scores: { starStructure: number; clarity: number; relevance: number }
    feedback: string
  }>
  codingQuestion: {
    title: string
    description: string
    examples: Array<{ input: string; output: string; explanation?: string }>
    constraints: string[]
    topic: string
    expectedComplexity: { time: string; space: string }
  } | null
  codingAnswer: string
  codingAnalysis: {
    scores: {
      understanding: number
      approach: number
      accuracy: number
      edgeCases: number
      communication: number
    }
    feedback: string
  } | null
}
```

### Interview Page State Machine

```
phase:          'behavioral' | 'transition' | 'coding' | 'done'
behavioralIndex: 0 | 1 | 2
status:         'idle' | 'recording' | 'recorded' | 'processing'
```

## 8. Design System

### Principles

The interface must feel like a tool made by a careful designer — restrained, typographic, calm. Candidates are already anxious; the UI should not add noise. Every visual choice earns its place.

Explicit constraints:
- No Inter font
- No pill-shaped buttons (`rounded-full` forbidden on interactive elements)
- No gradients
- No glassmorphism
- No box shadows as decoration
- No icon libraries (Lucide, Heroicons, Font Awesome)
- No emojis in UI text

### Typography

| Role | Font | Notes |
|---|---|---|
| Display / headings | Instrument Serif | Editorial weight; signals craft, not startup |
| Body / UI | DM Sans | Clean, readable, not Inter |
| Code / transcripts | JetBrains Mono | Contextually appropriate for technical content |

### Color Palette

| Token | Hex | Use |
|---|---|---|
| `background` | `#0A0A0A` | Page background |
| `surface` | `#141414` | Cards, panels |
| `border` | `#262626` | Default borders |
| `border-strong` | `#3A3A3A` | Hover / focus borders |
| `text` | `#F2EFE8` | Primary text — warm white, not pure |
| `muted` | `#737373` | Secondary text, labels |
| `accent` | `#D4C9B8` | Warm cream — highlights only, never backgrounds |
| `danger` | `#C84B4B` | Recording active state |

### Border Radius

All Tailwind radius values overridden to 2–4px. `rounded-full` reserved exclusively for the recording pulse dot.

### Buttons

- **Primary:** filled text-color background, 2px border-radius, no shadow
- **Secondary:** transparent background, border, same radius
- **Tertiary actions:** plain underlined text links — no border, no background

### Score Display

Monospaced tick bars, not gradient progress indicators:

```
Communication     ████░    4 / 5
STAR Structure    ███░░    3 / 5
Relevance         █████    5 / 5
```

Overall score displayed as a large serif numeral. Verdict displayed as a small-caps bordered label — no pill shape.

### Icons

Three custom inline SVGs (mic, stop square, arrow right). All other actions expressed as plain text: `RECORD`, `STOP`, `SUBMIT`, `RE-RECORD`, `BEGIN`, `×`.
